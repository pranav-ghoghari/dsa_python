# Work Habits

## Effort Pattern

- The student is willing to attempt real code and answer reflection prompts.
- The student explicitly asked not to be rescued too early.

## Helpful Structure

- One task at a time
- A visible success condition
- Short feedback loops using tests and review
- Concrete next steps instead of a broad backlog
- A clear post-task cleanup routine that preserves finished evidence instead of wiping files
- A short fundamentals explanation first when the next task uses a format the student has not practiced yet
- A reusable lesson shelf in the repo so taught concepts can be reread without rebuilding them from chat memory

## Avoidance Risks

- Explaining instead of coding
- Asking for theory before a real attempt
- Losing momentum when syntax gets frustrating

## Comments As A Tool

- Writing short comments can expose reasoning and help bridge from natural language into code.
- Use real `#` comments, not triple-quoted strings acting like comments.
- Writing input/output examples as comments is a useful intermediate step before formal unit tests.

## Motivation And Pacing

- The student's self-reported laziness should be treated as a planning constraint, not a judgment.
- External structure still appears useful for maintaining momentum and follow-through.
- The student prefers explicit repo cleanup after a finished task so the next step feels clear.
